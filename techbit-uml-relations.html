UML Relationships

  1. Dependency

    • [A] depends on [B] for certain function
    • weakest relationship
    • indicates loose coupling between two classes
    		 
    classA objA;
ClassB objB;
objB = SomeFunc( objA );
// or
class ChecksumValidator raises InvalidChecksumException

  2. Association (USES)

    • [A] uses [B]
    • stronger relationship than Dependency
    • indicates one or two way communication
    • direction of communication indicated by an arrow
    • no-arrow means two-way navigation
    • can be a named-association or unnamed
    • example
    		 
    car.addPassenger( personA );
...
car.dropPassenger( personA );
// or

class B {
  ...
};

class A {
  B * b_ptr;
};

  3. Aggregation (PART-OF / HAS-A)

    • [A] is "part of" [B]
    • however, [B] can exist without instantiating [A]
    • can be bi-directional (like Association)
    • If the "containing-object" is destroyed, the "contained-object" is destroyed with it
    		 
    class A {
   ...
};

class B {
  A * pA;
};

  4. Composition (HAS-A)

    • [A] is "part of" [B]
    • howeve, [B] cannot exist without instantiating [A]
    • is a STRONGEST relationship
    • Lifetime association and exclusive containment
    • If the "containing-object" is destroyed, the "contained-object" is destroyed with it
    		 
    class Human {
  class Hand { ... };
};
Human aPerson;

  5. Generalization (IS-A)

    • [A] "is-a-superclass-of" [B]
    • [A] "is-a-generalization-of" subclass-[B]
    		 
    class A {
  ...
};

class B : public A {
  ...
};

  6. Realization (EXTENDS)

    • [A] "implements interfaces offered by" [B]
    • Implementer provides concrete methods.
    		 
    class A {
  void draw() = 0;
};

class B : class A {
  void draw() {
    ...
 }
};
